# Experiment 001 — Live-model pilot (cloud path)

First pilot with **real LLM calls** and the **real SWE-bench images**, on a
**Daytona cloud sandbox**. Supersedes the environmental constraints of the
no-model instrument validation in [`../PILOT.md`](../PILOT.md), which ran on
*reconstructed* testbeds because that sandbox blocked every image-blob CDN and
`openrouter.ai`.

Design and research questions: [`../../DESIGN.md`](../../DESIGN.md).

## What this run answers

- **Q1** — do SWE-bench tasks run cleanly under Harbor with the chosen agent,
  **in the cloud, against the real images**?
- **Q2** — are regressions extractable at **PASS_TO_PASS granularity** from the
  **cloud** path? (*"Item 2 is the whole experiment."*)
- **Fidelity** — did last session's reconstructed testbeds produce the same
  numbers as the real images?

## Verdicts

**Q1 (cloud) — YES, with two documented hazards.** 7/7 trials built, ran and
graded on Daytona from the real `swebench/sweb.eval.x86_64.*` images. No local
Docker exists on this host and none was needed: Daytona builds a snapshot from
the task's Dockerfile server-side. 5/7 trials completed with **0 Harbor
exceptions**. The 2 failures were *not* model failures and are diagnosed below
(a litellm packaging bug, and an agent that killed itself); both were proven,
not inferred.

**Q2 (cloud) — YES.** Every trial, including both failures, returned
`verifier/report.json` with `tests_status.{FAIL_TO_PASS,PASS_TO_PASS}.{success,
failure}` as **lists of individual test names**. The verifier is isolated from
the agent phase and ran even when the agent phase raised. `harbor-report`
parsed the cloud job layout **without modification**.

**Fidelity — the reconstruction was faithful (for 15098).** See
[Reconstruction fidelity](#reconstruction-fidelity).

The headline live-model finding: on `django__django-15098` the model produced a
**plausible but over-broad fix that silently regressed a hidden PASS_TO_PASS
test** — the exact failure mode Experiment 001 exists to measure, this time from
a real model rather than a synthetic saboteur. It is reported with a caveat (the
trial died mid-iteration; see below), so it is **not** a clean capability
measurement.

## Hazard 1 — litellm packaging bug (an evil measurement artifact)

**Symptom.** The first live run burned 7m19s, returned reward 0, and produced a
report identical to `nop`. Superficially: "the model is incompetent."

**Reality: no model call was ever made.** Proven, not inferred:

- The trajectory contains roles `['system', 'user', 'exit']` — **no assistant
  message** — with `exit_status: ModuleNotFoundError`, `instance_cost: 0.0`.
- OpenRouter `usage` stayed at exactly its pre-run baseline (`2.8957e-05`).
- Reproduced in-sandbox: `litellm.completion(...)` **without** `tools` and with a
  **dummy** key reached OpenRouter and returned a genuine
  `{"error":{"message":"User not found.","code":401}}`. So sandbox egress to
  `openrouter.ai` works and `--ae` key plumbing is not at fault.

**Root cause.** `mini-swe-agent` v2 sends every request with `tools=[BASH_TOOL]`.
In `litellm` 1.92.0, `completion()` eagerly imports the MCP gateway handler
whenever `tools` is truthy (`litellm/main.py:4864`):

```python
skip_mcp_handler = kwargs.pop("_skip_mcp_handler", False)
if not skip_mcp_handler and tools:
    from litellm.responses.mcp.chat_completions_handler import acompletion_with_mcp
    ...
    if LiteLLM_Proxy_MCP_Handler._should_use_litellm_mcp_gateway(tools=tools_for_mcp):
```

That import reaches `litellm.proxy.litellm_pre_call_utils`, which does
`from fastapi import HTTPException, Request`. `fastapi` ships only in litellm's
`proxy` extra, so a plain `uv tool install mini-swe-agent` lacks it. The import
is unconditional even though the gateway is only *used* when MCP tools are
present — which they never are here. Every query therefore raised
`ModuleNotFoundError`; `mini-swe-agent` retried with exponential backoff
(4/4/4/8/16/32/60s) until the agent timeout, then exited non-zero. Harbor still
ran the verifier, which faithfully graded an untouched testbed.

**Why this matters beyond a bugfix.** The failure is silent and *directionally
biased*: it looks exactly like a competent harness measuring an incompetent
model. Had this landed in the real 4-condition run — or in only some arms — it
would have manufactured a difference between conditions out of a packaging
detail. Any arm reporting a suspiciously `nop`-like result must be checked for
`api_calls == 0` before it is believed.

**Fix.** [`ai_benchmark.live_agents.MiniSweAgentLitellmProxy`](../../../../src/ai_benchmark/live_agents.py)
subclasses Harbor's `MiniSweAgent` and reinstalls the tool `--with
"litellm[proxy]"`, then asserts `import fastapi, orjson` **at install time** so a
recurrence fails loudly instead of degrading into a retry loop. Covered by
[`tests/test_live_agents.py`](../../../../tests/test_live_agents.py).

`--with fastapi` alone is **not** sufficient (the chain next demands `orjson`);
the packaging extra is pinned rather than an ad-hoc module list.

## Hazard 2 — the agent killed itself (`pkill -f`)

On 15098 the patched agent worked for ~51 steps, then ran:

```
cd tshi && python manage.py runserver 8001 & sleep 5 && pkill -f runserver
```

Harbor launches the agent as
`mini-swe-agent --yolo --model=... --task='<the full issue text>' ...`. The
15098 issue text **contains the word "runserver"** (`grep -c` → 1). `pkill -f`
matches against the *full command line*, so the agent's own process matched and
was SIGTERM'd. Harbor surfaced this as `NetworkConnectionError` (the exec
channel dropped); `result.json` records `exit 143` (= 128 + 15, SIGTERM).

This is **not** model incompetence and **not** a network problem. It is a
structural hazard of passing task text in `argv`: any agent that runs
`pkill`/`pgrep -f` on a term appearing in its own task statement can kill itself.
16315 and 16429 contain no `"runserver"` and both completed cleanly, which is
consistent with (though does not by itself prove) the task-text-specific
explanation.

Consequence for the experiment: this failure mode is **correlated with task
text**, so it is not neutral noise across arms. It needs an explicit mitigation
before the real run.

## The live-model regression (15098), and its caveat

The model's final edit versus gold:

```python
# model: unlimited dash-separated segments
language_code_prefix_re = _lazy_re_compile(r'^/([a-zA-Z0-9]+(?:-[a-zA-Z0-9]+)*)(/|$)')
# gold:  at most two
language_code_prefix_re = _lazy_re_compile(r'^/(\w+([@-]\w+){0,2})(/|$)')
```

The model's regex accepts arbitrarily many `-` segments, so `/de-simple-page-test/`
parses as a language code and `test_page_with_dash` — a hidden PASS_TO_PASS test —
fails. (It also silently drops `@` and `\w`/underscore support.) This is a
textbook "looks like a reasonable fix, silently regresses something else" case,
surfaced **by name** at P2P granularity.

**Caveat — do not over-read this.** The trial was killed at step ~51 of an
**unbounded** run (Hazard 2), so the verifier graded a mid-iteration state, not a
patch the model considered finished. Its own FAIL_TO_PASS tests were still
failing (0/2), i.e. it had not solved the task either. This demonstrates the
**instrument** captures live-model regressions; it is **not** a measurement of
qwen3-coder's regression rate. n=1, one instance, one seed.

## Agent configuration (experiment parameters)

These were needed to run at all and **must be held constant across conditions**:

| Parameter | Value used | Default | Why |
| --- | --- | --- | --- |
| `cost_limit` | `1.0` (via `--ak cost_limit=1.0`) | Harbor forces `"0"` = **unlimited** | Budget guard. See the caveat below. |
| `step_limit` | **unset** (= `0`, unlimited) | mini.yaml `step_limit: 0` | Not set this session — a **known gap**, see below. |
| agent timeout | `3000s` (task.toml) | task default | Only real bound on an unbounded run. |
| verifier timeout | `3000s` (task.toml) | task default | — |
| env resources | `cpus=1`, `memory=4G`, `storage=10G` | task.toml | — |

Two findings here matter more than the values:

1. **Stock Harbor + mini-swe-agent runs with no step and no cost bound.**
   mini-swe-agent's packaged `mini.yaml` defaults to `step_limit: 0` (unlimited)
   and `cost_limit: 3.`, but Harbor's `MiniSweAgent.CLI_FLAGS` overrides
   `--cost-limit` to `"0"`, which is *also* unlimited. The only bound is
   wall-clock. That is a **validity problem**: with an unbounded step budget the
   effective number of steps becomes a function of provider latency, so arms and
   models are not comparable. `step_limit` must be set explicitly and held
   constant in the real experiment. It was not set here, which is part of why the
   15098 agent had room to wander into starting a dev server at step 51.

2. **`cost_limit` is not a trustworthy budget guard.** It is enforced against
   mini-swe-agent's *own* cost estimate, which under-reported real OpenRouter
   billing by **~8x** in this session (agent-reported $0.0325 total across the
   three live runs vs. ~$0.286 actually billed). Budget must be enforced
   provider-side (the OpenRouter key limit) — the in-agent number is an estimate,
   not an accounting record.

## Reproduction

Prerequisites: `uv`, this repo, an `OPENROUTER_API_KEY` and a `DAYTONA_API_KEY`
in the environment (`.env` at repo root is gitignored). **No Docker daemon
needed.**

```bash
uv sync   # harbor[daytona] == 0.18.0 + ai_benchmark
```

Live-model arm (patched agent — the unpatched one silently makes 0 model calls):

```bash
uv run harbor run -d swebench-verified@1.0 -i 'django__django-15098' \
  -a ai_benchmark.live_agents:MiniSweAgentLitellmProxy \
  -m openrouter/qwen/qwen3-coder -e daytona -n 1 -o jobs \
  --ae OPENROUTER_API_KEY="$OPENROUTER_API_KEY" \
  --ak cost_limit=1.0 --yes
```

Deterministic arms (`$0`, no model calls). `oracle`/`nop` are built in; the
custom `saboteur` needs the task dir explicitly (Harbor only auto-injects
`task_dir` for its own `oracle`):

```bash
uv run harbor run -d swebench-verified@1.0 -i 'django__django-15098' \
  -a oracle -e daytona -n 1 -o jobs --job-name real-15098-oracle --yes
uv run harbor run -d swebench-verified@1.0 -i 'django__django-15098' \
  -a nop -e daytona -n 1 -o jobs --job-name real-15098-nop --yes

TASK="$HOME/.cache/harbor/tasks/<hash>/django__django-15098"   # populated by any run above
uv run harbor run -d swebench-verified@1.0 -i 'django__django-15098' \
  -a ai_benchmark.pilot_agents:SaboteurAgent --ak task_dir="$TASK" \
  -e daytona -n 1 -o jobs --job-name real-15098-saboteur --yes
```

Summarise at PASS_TO_PASS granularity:

```bash
uv run ai-benchmark harbor-report jobs --json results.json --markdown results.md
```

`./jobs` is gitignored; the trimmed per-trial evidence lives in
[`trials/`](trials/).

## Secret hygiene

`--ae OPENROUTER_API_KEY=...` does **not** bake the value into Harbor's output.
Harbor stores the literal reference `"${OPENROUTER_API_KEY}"` in `config.json`
and interpolates from the environment at run time (verified: the stored string is
not the key, nor a prefix, substring, or 8-char overlap of it). The committed
configs here are additionally passed through a redactor, and every file under
this directory was scanned for the real key values and for `sk-or-` / `dtn_`
markers before commit.

## What the real 4-condition experiment now needs

Cleared by this pilot:

- Cloud path works end to end on real images; **no local Docker required**.
- P2P granularity survives the cloud path; `harbor-report` needs no changes.
- The reconstruction method is validated (15098), so last session's instrument
  validation is usable evidence.

Still required, in rough priority order:

1. **Set `step_limit` explicitly and hold it constant.** Currently unlimited;
   the effective budget is wall-clock, which makes conditions incomparable and
   let the 15098 agent wander into the self-kill. This is the highest-priority
   validity fix.
2. **Mitigate the `pkill -f` self-kill.** The hazard is correlated with task
   text, so it is not neutral across arms. Options: pass the task via a file
   rather than `argv`, or exclude tasks whose text collides with common process
   names. Needs a decision before the real run.
3. **Guard against silent zero-call arms.** Assert `api_calls > 0` (and
   non-empty assistant turns) per trial; a `nop`-shaped result must never be
   accepted at face value.
4. **Enforce budget provider-side.** `cost_limit` under-reports ~8x; use the
   OpenRouter key limit as the authoritative bound.
5. **Bake the agent into the snapshot.** 26-73s of per-trial install overhead
   across ~480 runs is avoidable.
6. **Pin images by digest**, not `:latest`.
7. **Repeat the fidelity check on 16315/16429** if their reconstructed numbers
   are to be relied on.
8. **Unchanged from `../PILOT.md`:** `swebench-live` is still absent from
   Harbor's registry (the DESIGN prefers a post-cutoff source for contamination
   resistance), and mutation-testing-based task filtering remains future work.

Not established by this pilot, and worth stating: **nothing here measures
adversarial review.** Three tasks, one seed, one model, no review arm. The
15098 live regression is a single contaminated observation. This pilot validates
the instrument on the cloud path; it produces no evidence about H1.
